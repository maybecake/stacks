# Project Context

## Purpose

A polyglot monorepo demonstrating a gRPC-first backend with a React frontend. The primary service is a greeting API (`HelloService`) that records greetings and exposes stats. The stack is intentionally polyglot ‚Äî the same domain logic is implemented in Go, TypeScript, and Python ‚Äî to exercise the hexagonal architecture pattern across runtimes.

## Tech Stack

### Frontend
- **React + TypeScript** ‚Äî strict mode, function components only
- **React Router v5** ‚Äî three routes: `/`, `/samples`, `/learner`
- **Vite** ‚Äî dev server on `:5173`; proxies `/api/*` to `:8080` (Connect RPC server)
- **Radix UI** ‚Äî primitive component library
- **Prettier** ‚Äî 2-space indent, 100-char line width

### Backend
- **Go** ‚Äî primary gRPC service (`service/simple/`); Bazel build
- **Python** ‚Äî alternative gRPC service (`service/simple/server.py`)
- **Protobuf / Connect RPC** ‚Äî all backend API surface defined in `api/protos/*.proto`; stubs generated by `buf generate`
- **Connect RPC** ‚Äî serves both Connect JSON protocol and gRPC from the same handler; no gateway needed

### Domain / Persistence
- **Hexagonal architecture** ‚Äî `api/domain/` for logic + port interfaces; `api/adapters/` for concrete implementations
- **Drizzle ORM** (TypeScript) ‚Äî schema management and migrations; schema in `api/db/schema.ts`
- **sqlc** (Go) ‚Äî type-safe Go structs generated from Drizzle migration SQL
- **psycopg v3** (Python) ‚Äî parameterised queries directly

### Build
- **Bazel** ‚Äî builds Go service and proto stubs
- **pnpm** ‚Äî frontend package manager

## Architecture Patterns

### Hexagonal Architecture (Ports and Adapters)
All persistence and external I/O is accessed through port interfaces defined in `api/domain/`. Each language defines its own idiomatic interface (`GreetingStore` in Go, TypeScript, and Python). Concrete adapters live in `api/adapters/` ‚Äî currently `mock/` (no-op) and `postgres/`. Handlers (`api/hello.ts`, `service/simple/simple.go`, `service/simple/server.py`) are the composition root: they construct the adapter and inject it into the domain function.

```
api/
  domain/         # Pure logic + port interfaces (no transport, no persistence imports)
  adapters/
    mock/         # No-op implementations (default)
    postgres/     # Neon Postgres implementations (TypeScript: Drizzle, Go: sqlc, Python: psycopg)
  db/             # Drizzle schema + client (TypeScript only)
```

### Connect RPC Backend API
All backend API endpoints are RPC methods defined in `.proto` files and served via Connect RPC. Plain HTTP handlers are not used as the API transport (except for legacy `api/stats.ts` which is being migrated). Connect serves the Connect JSON protocol natively ‚Äî no gateway process is needed.

## Architecture Decision Records

### ADR-001: Port interface per language, not a shared contract
**Status:** Accepted (hexagonal-api-refactor)

Each language defines its own `GreetingStore` interface in its domain package. There is no cross-language contract because there is no cross-language runtime.

**Rejected:** Proto-defined storage interface ‚Äî overengineered; storage is internal to each service, not a network boundary.

---

### ADR-002: All new backend API endpoints are gRPC
**Status:** Accepted

New backend API surface is defined as gRPC RPC methods in `hello.proto` and implemented via generated stubs. The grpc-gateway translates these to HTTP/JSON for the frontend. No plain HTTP endpoints are added as API transport.

**Rejected:** Plain `net/http` handlers or REST routes ‚Äî these bypass the proto contract and don't integrate with the service mesh.

---

### ADR-003: grpc-gateway as the frontend bridge
**Status:** Superseded by ADR-009 (greeting-stats-api)

Originally specified grpc-gateway, but the project already uses Connect RPC throughout. See ADR-009.

---

### ADR-009: Connect RPC as the frontend bridge (replaces grpc-gateway)
**Status:** Accepted (greeting-stats-api)

Connect RPC (`connectrpc.com/connect` in Go, `@connectrpc/connect` in TypeScript) is the project's RPC framework. Connect handlers natively serve the Connect JSON protocol (HTTP/1.1 with `application/json`) without any additional gateway or annotation layer. The frontend calls new RPC methods via plain `fetch` with POST requests to `/api/hello/hello.HelloService/<MethodName>`. Proto stubs are generated by `buf generate` (not Bazel protoc) and committed to `gen/go/`, `api/.gen/`, and `frontend/src/gen/`.

**Rejected:** grpc-gateway ‚Äî requires `google/api/annotations.proto` imports, a googleapis Bazel dep, generated gateway shim code, and a second HTTP listener. All of this is redundant when Connect already provides HTTP/JSON for free.
**Rejected:** Envoy + grpc-web ‚Äî separate process, more ops complexity.

---

### ADR-004: Drizzle ORM (TS) + sqlc (Go) + psycopg (Python) for persistence
**Status:** Accepted (greeting-counter-db)

Drizzle defines the canonical schema in `api/db/schema.ts` and generates migration SQL. sqlc points at Drizzle's generated migrations to auto-generate type-safe Go structs, eliminating manual schema drift. Python uses psycopg v3 with parameterised queries.

Migrations run at Vercel build time (`pnpm db:migrate` prepended to `buildCommand`) ‚Äî a failed migration blocks the deploy rather than deploying against a wrong schema.

**Rejected (Go):** Raw pgx ‚Äî requires manually maintaining Go structs that mirror the DB schema.
**Rejected (Python):** psycopg2-binary ‚Äî legacy adapter; psycopg v3 is the maintained successor.
**Rejected (Python):** SQLAlchemy ‚Äî full ORM is overkill for two queries in an adapter.

---

### ADR-005: AIP-132 cursor-based pagination for list RPCs
**Status:** Accepted (greeting-stats-api)

List RPC request messages include `int32 page_size` and `string page_token`. Response messages include the result list and `string next_page_token`. The server encodes the cursor as base64 of `{"offset":N}` ‚Äî opaque to callers. Default page size: 20. Maximum: 100 (clamped server-side).

**Rejected:** `int32 page` (page number) ‚Äî AIP explicitly discourages page numbers because they break under inserts.
**Rejected:** Unpaginated `GetStats` bundling both lists ‚Äî names can grow unbounded; pagination is required from the start.

---

### ADR-006: Two separate list RPCs, not one aggregate
**Status:** Accepted (greeting-stats-api)

`ListGreetingTypeStats` and `ListGreetedNames` are independent RPC methods with their own request/response message pairs. Each list is independently pageable.

**Rejected:** Single `GetStats` RPC returning both arrays ‚Äî pagination becomes ambiguous when cursors for greeting types and names are independent.

---

### ADR-007: GreetingStore interface extended, not split
**Status:** Accepted (greeting-stats-api)

Stats query methods (`ListGreetingTypeStats`, `ListGreetedNames`) are added directly to the existing `GreetingStore` interface rather than introducing a separate `StatsReader` interface.

**Rejected:** Separate `StatsReader` port ‚Äî premature CQRS split at this scale; adds interface proliferation for two read methods.

---

### ADR-010: Server handlers are Vercel serverless functions ‚Äî no persistent server process
**Status:** Accepted

The project deploys to Vercel. Each Connect RPC handler (TypeScript) runs as a Vercel serverless function. There is no long-running server process to manage, start, or keep alive. Infrastructure concerns like port binding, process supervision, and graceful shutdown do not apply to the TypeScript service layer.

**Consequence:** Go and Python service implementations (`service/simple/`) are development/exploration artifacts only. They are not deployed to production. Any new API surface that needs to be production-accessible must be implemented in the TypeScript handler layer (`api/hello.ts`, etc.) as a Vercel function.

**Rejected:** Running a persistent Go or Python gRPC server alongside Vercel ‚Äî introduces ops complexity (process management, health checks, networking) that Vercel's serverless model eliminates entirely.

---

### ADR-008: Handlers are the composition root
**Status:** Accepted (hexagonal-api-refactor)

Each handler file constructs the concrete adapter and injects it into the domain function. No global registry or DI container is used. Swapping from mock to Postgres adapter is a single constructor change in the handler.

**Rejected:** Global DB singleton imported by adapter ‚Äî hides the dependency and makes testing harder.
