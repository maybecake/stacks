## ADDED Requirements

### Requirement: Frontend uses generated Connect client to call SayHello

The frontend SHALL provide a typed `sayHello(name: string): Promise<string>` function in `frontend/src/api/hello.ts` that uses the buf-generated `HelloService` Connect client (not raw fetch) and returns the `message` string from the response.

#### Scenario: Successful call

- **WHEN** `sayHello("Alice")` is called
- **THEN** the Connect client sends a `SayHello` request with `name: "Alice"` via the Vite proxy and resolves with the `message` string

#### Scenario: Network or server error

- **WHEN** the server returns an error response
- **THEN** the function rejects with a Connect `ConnectError` describing the failure

### Requirement: Generated TypeScript types match the proto

The frontend SHALL import protobuf message types from `api/gen/ts/` (generated by buf from `hello.proto`) and NOT define hand-written request/response shapes. The `HelloRequest` and `HelloResponse` types used by the client SHALL be the buf-generated types.

#### Scenario: Proto field renamed

- **WHEN** a field in `hello.proto` is renamed and `buf generate` is re-run
- **THEN** the TypeScript compile step fails at the usage site, making the drift visible immediately

### Requirement: useHello hook manages async state

The frontend SHALL provide a `useHello(name: string)` React hook in `frontend/src/hooks/useHello.ts` that calls `sayHello` and exposes `{ data, loading, error }` state.

#### Scenario: Loading state while request is in flight

- **WHEN** the hook is called and the request has not yet resolved
- **THEN** `loading` is `true` and `data` is `null`

#### Scenario: Successful response

- **WHEN** the request resolves successfully
- **THEN** `loading` is `false`, `data` contains the message string, and `error` is `null`

#### Scenario: Error response

- **WHEN** the request rejects
- **THEN** `loading` is `false`, `error` contains the error, and `data` is `null`

#### Scenario: Re-fetches when name changes

- **WHEN** the `name` argument to `useHello` changes
- **THEN** the hook re-fetches and updates state accordingly

### Requirement: Prototype section provides interactive API testing

The frontend SHALL include a dedicated "Prototype" section on an existing page containing:
- A text input for the `name` parameter
- A "Send" button that triggers the `SayHello` request
- A read-only textarea that displays the response message (or error text)

The section SHALL use the `useHello` hook and allow the user to change the input and re-submit without reloading the page.

#### Scenario: User submits a request

- **WHEN** the user types a name into the input and clicks "Send"
- **THEN** the textarea shows a loading state while the request is in flight, then displays the returned message on success

#### Scenario: Server returns an error

- **WHEN** the request fails
- **THEN** the textarea displays a human-readable error message

#### Scenario: User changes the input and re-submits

- **WHEN** the user edits the name field and clicks "Send" again
- **THEN** a new request is made and the textarea updates with the new response

#### Scenario: Empty name input

- **WHEN** the user submits with an empty name field
- **THEN** the "Send" button is disabled or the request is not sent
